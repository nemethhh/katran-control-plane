# E2E Tests for Feature Parity — Design Spec

**Goal:** Cover all 10 features from the feature-parity implementation plan with E2E tests. API-level tests for all features, traffic-level tests where the BPF/network topology supports it.

**Features under test:** Source routing, inline decap, QUIC server ID, health checking, LRU management, down reals, extended stats, encap source IP, feature flags, extended Prometheus metrics.

---

## Infrastructure Changes

### New Docker Containers

Add to `docker-compose.e2e.yml`:

**backend-3** (`10.200.0.22` / `fd00:200::22`)
- Clone of backend-1/2. Same Dockerfile.backend, same entrypoint.
- Purpose: Source routing traffic target. Source route rules redirect traffic from test-client's CIDR to backend-3 specifically, proving the LPM route took effect (as opposed to CH ring distribution).
- Environment: `BACKEND_NAME=backend-3`, `BACKEND_ADDR=10.200.0.22`, `VIP_ADDR=10.200.0.10`, `VIP_ADDR6=fd00:200::10`.
- Must also accept VIPs used by other test files on tunl0 (multiple VIP addresses on tunl0).

**hc-target** (`10.200.0.30` / `fd00:200::30`)
- Runs `hc-target-server.py` — a raw-socket IPIP packet capture + HTTP API.
- Privileged with `NET_RAW` capability for raw socket access.
- Purpose: Receives HC probes generated by `healthchecking_ipip.bpf.o` on TC egress. Exposes captured probes via HTTP so tests can verify probes arrived with correct src/dst IPs.
- Dockerfile: Use `Dockerfile.backend` (has iproute2, tcpdump, python3). Copy `hc-target-server.py` alongside other scripts.

### LB Entrypoint Changes (`lb-entrypoint.sh`)

After loading the XDP balancer program:

1. **Load HC TC-BPF program:**
   ```bash
   tc qdisc add dev eth0 clsact
   tc filter add dev eth0 egress bpf direct-action \
     obj /app/katran-bpfs/healthchecking_ipip.bpf.o sec tc
   ```

2. **Pin HC maps** (from TC program):
   ```bash
   for map in hc_key_map hc_reals_map hc_ctrl_map hc_pckt_srcs_map \
              hc_pckt_macs hc_stats_map per_hckey_stats; do
     bpftool map pin name "$map" "$PIN_PATH/$map"
   done
   ```

3. **Pin additional XDP maps** (feature maps from balancer.bpf.o):
   ```bash
   for map in lpm_src_v4 lpm_src_v6 decap_dst server_id_map pckt_srcs \
              reals_stats lru_miss_stats quic_stats_map \
              decap_vip_stats server_id_stats; do
     bpftool map pin name "$map" "$PIN_PATH/$map" 2>/dev/null || true
   done
   ```

   Use the same program-ID cross-reference pattern as the existing map pinning to avoid stale map collisions.

4. **Environment variables for feature config:**
   ```
   KATRAN_FEATURES=src_routing,inline_decap,direct_healthchecking
   KATRAN_TUNNEL_BASED_HC=true
   ```

### Backend Entrypoint Changes (`backend-entrypoint.sh`)

Support multiple VIP addresses on tunl0. Backends currently add only `VIP_ADDR/32` and `VIP_ADDR6/128`. Add support for `EXTRA_VIP_ADDRS` env var:
```bash
for extra_vip in ${EXTRA_VIP_ADDRS:-}; do
    ip addr add "${extra_vip}/32" dev tunl0 2>/dev/null || true
done
for extra_vip6 in ${EXTRA_VIP_ADDRS6:-}; do
    ip -6 addr add "${extra_vip6}/128" dev ip6tnl0 nodad 2>/dev/null || true
done
```

This lets traffic tests use unique VIP addresses per test file while backends accept all of them.

### New Script: `hc-target-server.py`

Raw-socket IPIP capture + HTTP API.

- **Capture thread:** Opens `AF_PACKET` socket on eth0. Filters IPIP packets (IP protocol 4). For each captured packet, parses outer IP header (src, dst) and inner IP header (src, dst, proto). Stores in a thread-safe list with timestamp.
- **HTTP server** (port 8080):
  - `GET /health` → `{"status": "healthy"}`
  - `GET /probes` → `{"probes": [{"outer_src": "...", "outer_dst": "...", "inner_src": "...", "inner_dst": "...", "proto": 6, "timestamp": 1234567890.0}, ...], "count": N}`
  - `POST /probes/reset` → Clears captured probes, returns `{"status": "reset"}`
- Uses `struct` module to parse IP headers (no scapy dependency needed).
- DualStack HTTP server on `[::]` like `backend-server.py`.

### Backend Server Extension (`backend-server.py`)

Add endpoint for encap source IP verification:
- `GET /tunnel-info` → Reports the outer source IP of the last IPIP-encapsulated packet received on tunl0.
- Implementation: A background thread sniffs on `tunl0` via raw socket, recording the most recent outer source IP. The `/tunnel-info` endpoint returns `{"outer_src": "10.200.0.10", "last_seen": 1234567890.0}`.

**Helper functions** (used across test files):
```python
def make_vip_id(address, port, protocol="tcp"):
    return {"address": address, "port": port, "protocol": protocol}

def setup_vip(api_client, address, port=80, protocol="tcp"):
    resp = api_client.post("/api/v1/vips", json={"address": address, "port": port, "protocol": protocol})
    assert resp.status_code in (201, 409)
    return resp.json() if resp.status_code == 201 else None

def teardown_vip(api_client, address, port=80, protocol="tcp"):
    api_client.post("/api/v1/vips/remove", json={"address": address, "port": port, "protocol": protocol})

def add_backend(api_client, vip_addr, backend_addr, port=80, protocol="tcp", weight=100):
    resp = api_client.post("/api/v1/backends/add", json={
        "vip": {"address": vip_addr, "port": port, "protocol": protocol},
        "address": backend_addr, "weight": weight,
    })
    assert resp.status_code in (201, 409)
    return resp.json() if resp.status_code == 201 else None

def remove_backend(api_client, vip_addr, backend_addr, port=80, protocol="tcp"):
    api_client.post("/api/v1/backends/remove", json={
        "vip": {"address": vip_addr, "port": port, "protocol": protocol},
        "address": backend_addr,
    })

def send_request(vip_addr, port=80, timeout=5.0):
    url = f"http://[{vip_addr}]:{port}/" if ":" in vip_addr else f"http://{vip_addr}:{port}/"
    resp = httpx.get(url, timeout=timeout)
    resp.raise_for_status()
    return resp.json()

def send_requests(vip_addr, count=20, port=80, timeout=5.0):
    results = []
    for _ in range(count):
        try:
            results.append(send_request(vip_addr, port=port, timeout=timeout))
        except Exception:
            results.append(None)
    return results

def wait_for_condition(fn, timeout=10, interval=1):
    """Poll fn() until it returns truthy, or raise after timeout."""
    deadline = time.time() + timeout
    while time.time() < deadline:
        result = fn()
        if result:
            return result
        time.sleep(interval)
    raise TimeoutError(f"Condition not met within {timeout}s")

def parse_metric_value(content, metric_name, labels=None):
    """Parse a Prometheus metric value from text format."""
    import re
    if labels:
        label_parts = [f'{k}="{v}"' for k, v in sorted(labels.items())]
        label_str = ",".join(label_parts)
        pattern = rf"{metric_name}\{{{label_str}\}}\s+(\d+(?:\.\d+)?)"
    else:
        pattern = rf"^{metric_name}\s+(\d+(?:\.\d+)?)$"
    match = re.search(pattern, content, re.MULTILINE)
    return float(match.group(1)) if match else None
```

### Test Execution Order

Tests execute in three phases:

**Phase 1 — API CRUD** (no traffic dependency):
- test_feature_flags.py
- test_src_routing.py::API tests
- test_decap.py::API tests
- test_quic.py::API tests
- test_health_check.py::API tests
- test_down_reals.py::API tests
- test_encap_src_ip.py::API tests

**Phase 2 — Traffic** (generate traffic, verify routing):
- test_src_routing.py::Traffic tests
- test_down_reals.py::Traffic tests
- test_encap_src_ip.py::Traffic tests
- test_health_check.py::Traffic tests (HC probes to hc-target)
- test_decap.py::Traffic tests
- test_quic.py::Traffic tests

**Phase 3 — Stats & metrics** (traffic has populated BPF maps):
- test_lru.py (search/list/purge on populated LRU)
- test_stats.py (all stat endpoints)
- test_prometheus_extended.py (new Prometheus counters)

Ordering within a file is managed by class grouping. Cross-file ordering is not enforced by pytest (each file sets up its own state). Phase 3 tests that need populated LRU/stats create their own traffic within their setup fixtures.

| Test | Setup | Assertion |
|------|-------|-----------|
| test_hc_probes_generated | Configure HC fully: set src IP (LB's IP), src MAC (LB's gateway MAC or known MAC), dst MAC (hc-target's MAC), add HC key for VIP, add HC dst (somark→hc-target IP), set interface. HC probe generation: The HC BPF program runs on TC egress. When the kernel sends a packet with a socket mark (SO_MARK) matching a configured somark, the TC program intercepts it and rewrites src/dst to generate the HC probe. To trigger this, the control plane or a helper process on the LB container needs to send a packet with the appropriate SO_MARK. Alternatively, configure the HC program and verify via stats that it processed packets (if the LB container generates any marked traffic internally). The exact trigger mechanism depends on the BPF program behavior — if the HC BPF program scans all egress packets, normal API traffic may trigger stats.
